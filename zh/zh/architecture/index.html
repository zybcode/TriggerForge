    📐 TriggerForge 架构设计文档 (Architecture Specification)

    版本：v1.1-PRO 状态：已全量验收 (Approved) 定位：系统核心骨架与隔离模型规范，用于指导模块开发与核心代码实现。

    一、 系统架构全景

    TriggerForge 采用严格解耦的四层隔离架构模型，以确保系统在面对高并发文件事件、第三方插件崩溃或内存泄漏时，核心引擎仍能保持绝对的稳定与自愈能力。

│                        Looper (主循环)                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Sentry (监控哨兵)                           │
│        Watchdog 监听  →  ThreadPoolExecutor (并发分发)          │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                    ForgeCore (执行中心)                         │
│         为每个插件拉起独立子进程 (多进程沙盒隔离)                │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Clerk (输出管家)                           │
│         成功归档 + 失败归档 + Hexdump快照 + 诊断日志            │
└─────────────────────────────────────────────────────────────────┘

    二、 核心分层设计架构

    1. Looper (主循环层)

    • 核心职责：作为系统的最外层常驻守护进程，负责整个 TriggerForge 引擎的生命周期管理（启动、配置加载、优雅停机与全局信号捕获）。
    • 自愈机制：在捕获到未知的全局异常时，保障配置不丢失，并具备动态重启下层服务的能力。

    2. Sentry (监控哨兵层)

    • 核心职责：高并发事件监听与分发中心。
    • 技术实现：
    • Watchdog 监听器：采用非阻塞模式对配置的监控目录进行实时文件事件扫描。
    • ThreadPoolExecutor 线程池：当捕获到文件触发事件时，Sentry 不在主线程执行业务，而是将事件打包为任务，即时推入内部线程池进行并发分发，避免主循环因 I/O 阻塞而失去响应。

    3. ForgeCore (执行中心层)

    • 核心职责：插件的宿主环境与核心执行引擎。
    • 沙盒隔离机制（Sandbox Isolation）：
    • 多进程硬隔离：ForgeCore 严禁在主进程或线程池内部直接执行插件代码。对于每一个触发的插件任务，ForgeCore 会通过 runner.py 拉起一个独立的子进程。
    • 资源与崩溃防御：插件子进程拥有独立的内存空间与硬性超时熔断机制。若第三方插件发生内存泄漏、陷入死循环或突发崩溃（如段错误 Segfault），受损的仅为该子进程，ForgeCore 会在超时或崩溃后安全回收资源，核心引擎主干保持 100% 炸不烂。
    • 多版本与虚拟环境支持：支持在配置中显式指定 python_path，允许子进程动态调用完全隔离的外部虚拟环境（Venv），解决 legacy 插件与现代依赖的冲突问题。

    4. Clerk (输出管家层)

    • 核心职责：负责插件执行完毕后的收尾工作与状态物化。
    • 资产管理：
    • 流水归档：根据子进程返回的执行状态，自动将处理后的文件移动至 success_path（成功归档）或 error_path（失败归档）。
    • 诊断快照 (Hexdump)：当插件触发异常或故障时，Clerk 会自动抓取相关文件上下文的二进制 Hexdump 快照，连同标准错误流（Stderr）一同写入诊断日志，为 SRE 和运维团队提供零咨询排查证据。

    三、 防御性工程设计规范

    1. 架构即代码 (Architecture as Code)

    静态文档极易随着代码迭代而失效。TriggerForge 严禁手动修改配置说明文档。所有系统配置（如 watch_folders、concurrency、strategies 等）的元数据必须在 src/triggerforge/core/schema.py 中通过 Python 类型系统 + Pydantic Schema 进行自描述声明。通过运行 scripts/generate_docs.py 自动化抽取 Schema，实现 docs/configuration.md 的自动更新。

    2. 现代 Python 源码布局 (Src Layout)

    项目摒弃传统的嵌套包结构，严格采用 src/ 布局（配合 pyproject.toml 的 Hatch 构建后端）。代码主干位于 src/triggerforge/，有效防止本地调试时因嵌套路径污染 PYTHONPATH 而导致的导入隐式报错，确保打包为 Wheel 文件时路径绝对纯净。

    3. CI/CD 确定性消除 Flaky Tests

    为了确保在 GitHub Actions 等容器化或虚拟化 CI/CD 环境中管道的 100% 稳定，测试目录（tests/）下的集成测试严禁依赖操作系统的真实磁盘 I/O 事件监听。测试套件通过 tests/conftest.py 引入内存队列模拟文件系统（Mock Filesystem）。通过内存事件桩对系统事件进行确定性触发，彻底消除因操作系统事件延迟导致的“偶发性超时测试失败”。
