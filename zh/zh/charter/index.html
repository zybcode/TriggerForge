    📜 TriggerForge 项目章程 (Project Charter)

    版本：v1.1-PRO 状态：已全量验收 (Approved) 定位：明确 TriggerForge 的核心业务痛点、系统边界与技术愿景，作为项目启动的基石文档。

    一、 核心业务痛点与项目背景

    在现有的企业自动化流水线和数据处理场景中，通过监听目录文件事件来触发后续业务逻辑（如文件解压、格式转换、数据解析等）是非常普遍的需求。然而，现有的常规脚本和轻量工具在生产环境中面临三大致命痛点：

    1. “僵尸文档”与配置漂移：传统的系统配置与开发文档割裂，随着项目快速迭代，文档极易失效，最终变成误导新人的过期垃圾。
    2. “一炸全炸”的插件环境：第三方插件或遗留（Legacy）脚本通常编写质量参差不齐。当插件发生内存泄漏、陷入死循环或突发段错误（Segfault）崩溃时，往往会直接拖垮整个监控主进程，缺乏有效的故障隔离手段。
    3. Flaky Tests（不稳定测试）对 CI/CD 的污染：由于 Watchdog 等底层库依赖操作系统的文件系统事件监听，在 GitHub Actions 等容器化或虚拟化 CI/CD 环境中，经常因磁盘 I/O 延迟表现出极其不稳定的“偶发性超时失败”，极大地消耗了团队的维护精力。

    TriggerForge 正是为了彻底终结这些乱象而生。它从设计之初就拒绝做一个“能跑就行”的临时脚本，而是演进为一个具备企业级隔离能力、自动化文档生成、开箱即用部署方案的全生命周期工程产品。

    二、 技术愿景与核心目标

    TriggerForge 的核心愿景是打造一个“跑得稳、炸不烂、文档全、能协作、可开源”的现代事件驱动型目录编排引擎。

    1. 核心工程目标

    • 架构即代码（Architecture as Code）：利用 Python 类型系统与 Pydantic Schema 作为“活文档”的源头。严禁手动修改配置说明，所有字段元数据由系统配置 Schema 自动生成，消除文档漂移。
    • 多进程硬核沙盒隔离（Sandbox Isolation）：为每个插件的执行拉起独立的子进程，并施加严格的硬性超时熔断机制与资源边界，将故障风险完美锁死在子进程内部。
    • 云原生与 CI/CD 确定性：引入内存队列模拟文件系统的 Mock 机制，绕过容器环境下的真实磁盘 I/O，确保测试管道 100% 稳定通过。

    三、 系统边界与功能交付范围 (v1.1-PRO)

    1. 准予交付的核心功能 (In Scope)

    • 现代化源码布局：采用符合现代 Python 社区最佳实践的标准 src/ 布局与 pyproject.toml 构建配置（基于 Hatch 后端与 PEP 621 标准）。
    • 非阻塞并发事件分发：主循环（Looper）与监听哨兵（Sentry）协作，利用线程池（ThreadPoolExecutor）实现文件事件的并发分发，绝不阻塞主监控流。
    • 动态多虚拟环境支持：支持在配置中显式指定 python_path，允许子进程动态调用完全隔离的外部虚拟环境（Venv），完美兼容不同版本的历史插件。
    • 生产级运维闭环：提供开箱即用的 Linux Systemd 服务守护配置与基于 journald 的可观测性日志流追踪，同时在插件故障时自动抓取二进制 Hexdump 快照进行物化诊断。

    2. 明确排除的功能边界 (Out of Scope)

    • 网络 API 接口：v1.1-PRO 版本定位为纯粹的常驻本地守护进程，不适用且不提供任何基于 HTTP/RESTful 的网络接口，保持极简与纯粹。

    四、 商业与开源路线声明

    为了平衡社区开源协作的生态红利与企业级闭环的商业价值，TriggerForge 采用双重许可模式（Dual-licensing）：

    1. 开源免费版 (GPL v3)：针对个人学习、学术研究及开源社区使用，代码全面开源，受 GPL 协议的“传染性”保护，任何将其集成的分发产品必须同步开源。
    2. 商业授权版 (Commercial License)：面向希望将 TriggerForge 集成进闭源商业软件或 SaaS 平台的企业客户，提供免除 GPL 传染限制的独立商业许可、高阶插件定制及 SRE 级服务支持。
    3. 贡献者合规性：所有向本项目提交 Pull Request 的外部开发者，必须签署贡献者许可协议（CLA），以保障项目版权的完整性与双重许可的合法分发。
